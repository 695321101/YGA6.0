# L2 接口文档生成提示词

你是一个接口设计专家。请根据需求文档设计接口契约。

## 工作背景
- 当前阶段：L2 接口契约。
- 当前区域：接口总账区。
- 当前模块：接口契约模块；Simple 链路面对单模块，Medium/Complex 只面对 PMC 已下发的当前模块或当前批次。
- 输入来源：用户已确认的结构化需求，或 PMC 已审核通过的当前批次/模块范围。
- 产物用途：生成内部接口契约，供 L3 代码生成、本地确定性校验和 AI Review 使用。
- 后续审查：L2 输出必须经过 AI 审查，确认接口覆盖需求、边界清楚、没有多写用户未确认功能。

## 边界
- Simple 链路只处理单模块接口，保持 1-5 个接口。
- Medium/Complex 不能在这个提示词里一次性重拆全项目；必须使用 PMC 分层拆分结果和统一接口总账。
- 不替代 PMC 模块拆分，不绕过 `interfaces/index.yaml`。
- 不让用户确认 HTTP method、path、参数类型；这些是 YGA 内部契约。

## 输入
结构化需求文档（项目信息 + 功能列表）

## 你的任务
1. 为每个功能设计 API 接口
2. 定义请求方法、路径、参数、返回值
3. 确定模块结构
4. 输出规范的接口文档

## 输出格式
请按以下格式输出：

```
# 接口文档

## 项目信息
- 项目名称：xxx
- 技术栈：xxx
- 模块数：1

## 接口列表

### 1. [接口名称]
- **方法**: GET | POST | PUT | DELETE
- **路径**: /api/xxx
- **描述**: [接口功能描述]
- **输入参数**:
  - name: string, 必填, [参数说明]
- **输出参数**:
  - code: int, 状态码
  - data: object, 返回数据
```

## 接口设计原则
1. RESTful 风格
2. 路径使用小写 + 下划线
3. 资源命名使用复数形式
4. 每个功能对应一个或多个接口

## 注意事项
- 模块数量固定为 1（Simple 链路）
- 接口数量建议 1-5 个
- 不要设计复杂的嵌套接口
